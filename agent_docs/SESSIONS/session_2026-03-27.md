### 13:39 — ARCHITECTURE REVIEW
 **Task:** Evaluate the next starter kit RBAC direction using Supabase `user_roles`, triggers, and RLS  
 **Decision:** Feasible to prototype safely in the current Supabase project **if done additively** without replacing existing auth flow in one shot.  
 **Recommended approach:**
 1. Create a new canonical `user_roles` table without touching existing business tables first.
 2. Backfill current users into `user_roles` from existing metadata flags.
 3. Add a trigger so newly created auth users default to `member`.
 4. Rewrite registration/admin-creation code to write into the new model while keeping existing users intact.
 5. Apply RLS only to new tables or to carefully selected migrated tables after validation.
 6. Treat the current Supabase project as a migration sandbox and preserve all SQL/scripts so the setup can be reproduced cleanly in a fresh Supabase project later.
 **Key constraint:** Do **not** flip existing production-style tables to strict RLS until role backfill, policies, and app queries are fully tested; otherwise existing users can lose access unexpectedly.
 **Conclusion:** Yes, the new architecture is possible on the current Supabase setup **without messing with current users**, but only through a staged, additive rollout. It should also be documented as SQL migrations/scripts so a fresh Supabase project can be recreated cleanly later.
